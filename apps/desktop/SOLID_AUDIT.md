# Solid Desktop — Architectural Audit

A standing audit of `apps/desktop/src/**` against the React-skill principles
(`~/.claude/skills/react/`) and Cristian's first-principles set
(`~/.brain/principles/`). The React rules don't map 1:1 to Solid — Solid has
fine-grained reactivity, no re-renders, no `useEffect` waterfall — but the
**underlying principles** (derive don't sync, model state tightly, compose over
flags, name events not setters, etc.) apply unchanged.

This document is the **input** to a `/goal` that iterates until the violations
section is empty. Treat it as a living artifact: agents append findings, the
`/goal` strikes them off, the loop terminates when **Status** is `clean`.

## Stack reminder (read before auditing)

- **Solid 1.9** — fine-grained reactivity via `createSignal` / `createMemo` /
  `createEffect`. There is **no virtual DOM and no re-render**. The "don't
  derive with useEffect" rule from React translates to "don't `createEffect`
  to write a signal you could express as `createMemo`."
- **Effect** — services in `src/services/` are `Context.Tag`-based and live
  inside the `runtime` (`src/runtime.ts`). Async work flows through
  `Effect.runPromise` / `runFork`. Fibers are the cancellation primitive.
- **Electron** — IPC procedures defined in `src/procedures.ts`, cached/proxied
  via `src/ipc-cache/`.
- **Motion** — custom Solid-friendly motion layer in `src/motion/` wrapping
  `motion-dom`.

Solid-specific gotchas to fold into the audit lens:

- Destructuring props loses reactivity → use `props.x` directly or `splitProps`.
- `createMemo` is the right tool for derived values; `createEffect` is for
  side effects only (DOM mutation, subscriptions, IPC fire-and-forget).
- `Show` / `Switch` / `For` are the platform's primitives; ad-hoc ternaries
  with `() => x() && <Thing/>` work but lose readability past one level.
- A `createSignal` that's only ever written from one `createEffect` that reads
  another signal is almost always a derived value in disguise — kill the
  signal, use a memo.

## Audit dimensions

Each dimension below cites the principle(s) it's derived from and lists the
concrete questions to ask while reading code.

### A1 — State modeling (`make-impossible-states-unrepresentable`, react `state-modeling.md`)

- Are boolean clusters (`isOpen`/`isAnimating`/`isClosing`, `isLoading`/`isError`/`data`) modeled as discriminated unions with a `status` / `_tag` discriminant?
- Do fields live on the state they describe (`data` only on `success`, `error` only on `error`) — no `data?: T` floating on a parent?
- Are state machines named by lifecycle ('closed' | 'opening' | 'open' | 'closing'), not by flag?
- Do reducers / setters guard invalid transitions, or do they silently overwrite?

### A2 — Derive, don't sync (`derive-dont-sync`, react `state.md`)

- For every `createSignal`, ask: could this be a `createMemo` of existing state? If yes — it's syncing, not deriving.
- Any `createEffect(() => setX(f(y())))` is a syncing antipattern — collapse to `const x = createMemo(() => f(y()))`.
- Any `localStorage` / `settings-storage` write paired with a duplicated in-memory signal: is one a projection of the other, or are they independently mutable?
- Are computed visual states (e.g. drawer position, scroll offset) stored alongside their inputs?

### A3 — Composition over flags (`composition-over-flags`, react `composition.md`)

- Any component with 3+ boolean props that switch render branches → split into named variants.
- Any component whose body is `props.variant === 'x' ? ... : ...` → consider compound components with a shared context.
- Render-prop / `renderHeader` / `renderFooter` patterns — should be `children` or slots unless data flows parent→child.
- Mega-components (>400 LOC) with mixed concerns — `app.tsx` (1302), `bible-chapter-canvas.tsx` (774), `command-palette.tsx` (672) — list the concerns and propose splits.

### A4 — Name events, not setters (`name-events-not-setters`)

- Reducer actions / signal setter call-sites named after **what happened**
  (`bookOpened`, `chapterRequested`) not **what changed** (`setOpenBook`,
  `setChapterIndex`)?
- Service methods on Effect services: are they domain verbs or property-bag
  setters?
- Callbacks passed as props: `onSubmitted` / `onVerseSelected` vs `onChange` /
  `onClick` when something more specific is meant.

### A5 — Lifecycle / cancellation (`lifecycle-tasks-cancel-on-reentry`)

- Every `createEffect` / `onMount` that spawns a fiber / subscription: does it
  cancel on re-run? Solid's `onCleanup` is the equivalent of React's effect
  cleanup return — verify every async spawn has one.
- `Effect.runFork` call sites: who interrupts the returned fiber when the
  context dies? `Fiber.interrupt` must be wired.
- IPC streams in `egw-ipc-client`, `prefetcher`, `search-service` — do they
  abort when the user navigates away mid-flight?
- Stale-result guard: when a fiber resolves, does it check the world it was
  spawned in still matches the world it resolves into?

### A6 — Small interface, deep implementation (`small-interface-deep-implementation`, `boundary-discipline`)

- Public exports of each `services/*.ts` file: count them. Anything exported
  that's only used inside the same file?
- Services that forward to another service without adding logic — collapse or
  delete the layer.
- Components that re-export their own internal types — usually a leak.
- `procedures.ts` surface: is each procedure carrying its weight, or are some
  thin wrappers around a single IPC call?

### A7 — Use the platform (`use-the-platform`, react `state.md` §Use the platform)

- URL / hash state: is reading-position / open-book / open-chapter in the URL,
  or hidden in a signal that resets on reload?
- `key` prop to reset subtrees on identity change — used where appropriate, or
  replaced with manual `createEffect(() => reset())`?
- `FormData` / native form submission vs controlled signal-per-input where
  forms exist.
- Storage: `localStorage` vs `settings-storage` adapter — is the adapter
  earning its keep, or is it just `JSON.parse(localStorage.getItem(...))`
  with extra steps?

### A8 — Impossible states (`make-impossible-states-unrepresentable`)

- Specific case from CONTEXT.md: `DrawerState = 'closed' | 'toc' | 'tocPlusLib'`
  — confirmed good. Find the **bad** equivalents: are there places still
  using paired booleans?
- Optional fields on top-level state objects — convert to discriminated
  variants where one optional field is meaningful only under a specific
  condition.
- `null | undefined` doubles in service return types: pick one. Effect's
  `Option` is preferred since the codebase is already in Effect.

### A9 — Effect runtime hygiene

- `Effect.runPromise` / `runFork` from inside Solid components — are they
  wrapped in `onCleanup`? Returning `Fiber.interrupt` to `onCleanup` should
  be the default.
- Service tag duplication: each service should be defined **once** as a
  `Context.Tag`. Grep for accidental re-declarations.
- Layers composed at the runtime root (`runtime.ts`) — services should not be
  layering themselves inside components.
- Synchronous `runSync` calls in component bodies — these block the renderer;
  every one needs justification.

### A10 — Solid-specific reactivity hygiene

- Props destructuring in component signatures (`({ foo }) => ...`) — breaks
  reactivity. Should be `(props) => props.foo` or `splitProps`.
- `createSignal` used where `createMemo` would suffice (single derived value
  written from one effect that depends only on other signals).
- `Show` / `Switch` / `For` used consistently — not `() => xs().map(...)`
  patterns that lose keying.
- Manual `untrack()` / `batch()` calls — every one needs a one-line comment on
  why (else it's premature optimization or a hidden reactivity bug).
- `createEffect` doing pure derivation (no DOM / no IPC / no setter to an
  unrelated signal) — convert to `createMemo`.

### A11 — Encode lessons in oxlint (`encode-lessons-in-structure`)

Sibling repo `~/Developer/personal/gent` proves out the pattern: a
`./lint/*.ts` file exports a `Plugin` (typed via `#oxlint/plugins`) registered
in `.oxlintrc.json` under `jsPlugins`, with rules listed under `rules` as
`"<plugin-name>/<rule-name>": "error"`. Gent ships 15 custom rules this way
(see `~/Developer/personal/gent/.oxlintrc.json` + `~/Developer/personal/gent/lint/no-direct-env.ts`).

We already have `oxlint 1.63.0` installed at the repo root. Adding a JS plugin
is a 3-step change:

1. Create `apps/desktop/lint/solid-audit.ts` exporting a `Plugin`.
2. Add `"jsPlugins": ["./apps/desktop/lint/solid-audit.ts"]` to
   `.oxlintrc.json` (or a desktop-scoped override).
3. List the rules under `rules` and set them to `"error"` (or `"warn"` for
   bedding-in).

Each rule for A11 should be a **structural fix for a recurring finding from
A1–A10** so the audit doesn't have to re-find them next time. Candidates:

| Rule slug                          | Catches                                                                                                                      | Sourced from |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ------------ |
| `solid/no-runpromise-then-set`     | `runtime.runPromise(...).then(setX)` — every A5/A9 race hole                                                                 | A5, A9       |
| `solid/runpromise-needs-cleanup`   | `Effect.runPromise` / `runFork` inside a Solid component scope without an `onCleanup` in the same scope                      | A5, A9       |
| `solid/no-effect-as-memo`          | `createEffect(() => setX(f(y())))` with a single setter call and no other side effect                                        | A2, A10      |
| `solid/no-paired-bool-state`       | sibling `createSignal<boolean>` + `createSignal<T \| null>` written together (paired-flag antipattern)                       | A1, A8       |
| `solid/no-double-nullable`         | function return type or service field of shape `T \| null \| undefined`                                                      | A8           |
| `solid/effect-service-no-setters`  | Effect service class with a method matching `^set[A-Z]` (force domain-verb naming)                                           | A4           |
| `solid/no-pushup-loader-component` | component whose entire body is `createEffect(() => props.setX(value))` (the StrongsLoader/MarginNotesLoader push-up pattern) | A2, A10      |
| `solid/component-max-loc`          | component file >500 LOC fails (forces splits; tune per-file overrides for the long tail)                                     | A3           |
| `solid/no-unused-export`           | re-validates motion/ipc-cache barrel exports against actual call sites                                                       | A6           |
| `solid/no-hand-rolled-debounce`    | calls matching `setTimeout(...)` inside a closure that clears a captured `Fiber` / `number` — replace with `Effect.debounce` | A7           |

Rules to defer (need type info or aren't high-signal yet): URL-state checker,
boolean-explosion detector beyond paired flags, automated event-name
suggestions.

**The rule:** every `/goal` pass that fixes ≥3 violations of the same shape
must add an A11 rule that would have caught them. The audit is then
_structurally_ clean for that shape, not just instance-clean.

See `apps/desktop/lint/PLAN.md` (created by this audit) for per-rule AST
shape sketches.

## The `/goal` loop

Once Violations is populated:

```
/goal fix every open row in apps/desktop/SOLID_AUDIT.md §Violations.
       for each row: read the file:line, apply the fix sketch (or improve on
       it), run `bun run typecheck && bun run test` from apps/desktop/, commit
       with `refactor(desktop): A{N}-{XX} — {short}`, strike the row through.

       AFTER fixing ≥3 rows of the same shape (same principle, same rough
       pattern), add an A11 oxlint rule from apps/desktop/lint/PLAN.md that
       would have caught them, run `bun run lint` to confirm clean, and
       commit as `feat(desktop/lint): A11-{slug} — {short}`. The rule is the
       structural fix; the row fixes are the migration.

       stop when every row is struck or you hit a row whose fix you can't
       complete without user input (then ask).
```

Termination: every row resolved AND every recurring shape encoded as an A11
rule → flip **Status** to `clean`. Re-running the audit later either keeps it
clean (the rules catch new violations at lint time) or surfaces new
dimensions to add.
